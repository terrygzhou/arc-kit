# LinkedIn Version: I Built ArcKit for Myself. Thank You for Using It.

## LinkedIn Article Body

I built ArcKit for myself.

That is the honest origin story.

I wanted a way to make architecture work less wasteful and more traceable. I wanted requirements, risks, ADRs, Wardley maps, business cases, procurement notes and assurance evidence to live together instead of drifting across documents, decks and half-remembered decisions.

I wanted a harness that made AI useful for architecture without letting it turn governance into a pile of plausible prose.

So I built the thing I needed.

I did not expect this many people to use it.

Over the last few weeks I looked at the public signals around ArcKit: GitHub, the website, search data, clone activity and the standalone assistant extension repos.

They are not perfect measures. ArcKit does not phone home. I cannot tell you how many teams have run a command or how many architecture decisions have been made with it, and I am comfortable with that. A governance toolkit should not quietly collect telemetry from the projects it helps.

But even from the public signals, one thing is now obvious.

ArcKit is no longer just my local working method.

People are using it. People are cloning it. People are forking it. People are installing the assistant-specific versions. People are searching for it by name. People are reading the docs, opening the guides, inspecting the skills, and coming back to the repository from the site.

That is both wonderful and slightly unreal.

As of 29 June 2026, ArcKit had:

- 2,039 GitHub stars
- 253 forks
- 197 GitHub releases
- 304 merged pull requests
- 197 closed issues

The main repo also saw this in GitHub's latest traffic window:

- 2,974 clone events
- 625 unique cloners
- 1,864 repository views
- 642 unique repository viewers

Clone events are not active users. They include repeat pulls, CI, local rebuilds, automation and people trying the project once.

But cloning is still different from browsing. It means someone put ArcKit on a machine.

That is the moment where a project stops being content and starts becoming a tool.

The assistant-specific repos surprised me too. In the same GitHub traffic window:

- arckit-codex had 594 clone events from 219 unique cloners
- arckit-gemini had 151 clone events from 64 unique cloners
- arckit-copilot had 114 clone events from 38 unique cloners
- arckit-opencode had 81 clone events from 42 unique cloners
- arckit-vibe had 81 clone events from 60 unique cloners

Do not read those as deduplicated users. The same person can clone more than one repo.

Read them as behaviour. People are pulling the version of ArcKit that fits the way they already work: Claude Code, Codex CLI, Gemini CLI, OpenCode, GitHub Copilot, Paperclip or Mistral Vibe.

That matters because ArcKit is not trying to be another AI app.

It is a harness for the places where architects already work.

The public site is doing more than I expected too. In the latest Cloudflare export, covering 30 May to 28 June 2026, arckit.org served:

- 95,631 requests
- 477 daily unique visitors on average
- 858 daily unique visitors on the busiest day
- 11.31 GB total data served
- 5.32 GB data cached

Cloudflare reports unique visitors by day, not as a deduplicated 30-day audience, so I treat that as a daily attention signal rather than a monthly user count.

Still, the shape matters. The site is getting steady daily attention. It is not just a one-day launch spike.

Search is showing the same thing.

The latest Google Search Console export recorded:

- 764 search clicks
- 9,874 impressions
- 7.74 percent click-through rate
- 9.65 weighted average position

The top query was simply "arckit", with 426 clicks and 1,943 impressions.

That is another strange moment for a project that started as a personal tool: people are searching for it by name.

The audience is wider than I expected too.

The UK is still first by search clicks, which makes sense. ArcKit started with a strong UK public-sector centre of gravity: GDS, Technology Code of Practice, Green Book, Orange Book, NCSC CAF, Digital Marketplace, assurance and procurement.

But the search data also shows meaningful attention from the United States, France, Australia, the Netherlands, India and Germany.

That fits what ArcKit has become.

The core method is general: make architecture decisions explicit, traceable and reviewable.

The overlays make it local.

I care about this because I have seen how much public money and delivery time gets lost when decisions are vague, evidence is scattered, and governance arrives too late to change anything.

ArcKit was my attempt to make the boring but necessary work faster without making it sloppier.

The toolkit drafts.

The architect judges.

That has always been the point.

AI can help assemble the evidence, structure the artefacts, enforce naming, connect requirements to decisions, check traceability and keep a project moving.

It cannot carry the accountability.

It cannot know the political context.

It cannot decide what risk a department should accept.

It cannot replace the judgement of someone who has to stand behind the recommendation.

So when I see clone activity, search demand and extension usage, I do not read it as "AI is replacing architects."

I read it as architects, engineers, product people and delivery teams looking for a better way to do the work they already know matters.

That is the part I am grateful for.

Thank you for trying ArcKit.

Thank you for starring it.

Thank you for cloning it.

Thank you for filing issues, opening pull requests, asking questions, poking at the weird edges, and adapting it to assistants I was not originally thinking about.

Thank you for treating enterprise architecture as something that can be made more open, more reproducible and more useful.

I built ArcKit for myself.

You made it real.
